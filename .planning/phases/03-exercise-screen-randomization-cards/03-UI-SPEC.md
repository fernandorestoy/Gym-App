## Color

Full palette established in Phase 1. Phase 3 adds no new color tokens.

| Role | Hex | CSS Variable | Phase 3 Element |
|------|-----|-------------|-----------------|
| Dominant (60%) | `#FFFEF1` | `--color-cream` | Card background, page background |
| Secondary (30%) | `#1A2742` | `--color-navy` | All text, card border at full opacity, button borders |
| Accent (10%) | `#82CDD8` | `--color-teal` | Card border (2px), focus rings, muscle group tag background (15% alpha) |
| Semantic — Regenerate | `#E58C8A` | `--color-dusty-rose` | Regenerate button background only |
| Semantic — tag | `#5BA08A` | `--color-sage` | Not used in Phase 3 (reserved for ≥18px bold text; no such element in Phase 3) |

Accent (`--color-teal`) reserved for in Phase 3:
1. Exercise card border: `2px solid var(--color-teal)`
2. Focus-visible outline on all interactive elements: `3px solid var(--color-teal)`, offset 3px
3. Muscle group pill background: `rgba(130, 205, 216, 0.15)` (teal at 15% alpha on cream)
4. SVG illustration accent color (passed as `accentColor` argument to `getSvg()`): use `#82CDD8`

Dusty Rose (`--color-dusty-rose`) reserved for:
- Regenerate button background only. Text on Dusty Rose: `--color-navy` (contrast ~4.5:1, passes WCAG AA for normal text).

Color rules:
- No gradients. No shadows. No 3D effects. (DSNG-05 — hard constraint)
- SVG illustration: `getSvg()` currently renders with a hardcoded `linearGradient` background (`#F0EBE3` → `#E4DDD4`) and Spanish labels "INICIO" / "FIN". Phase 3 must override: pass `accentColor = '#82CDD8'` to `getSvg()`. The gradient background in the SVG is inside the SVG element itself and does not violate the CSS no-gradient rule — it is acceptable as-is for Phase 3 (fixing the SVG internals is Phase 5 polish). The Spanish labels ("INICIO", "FIN") must be updated to "START" / "END" in `js/svgIllustrations.js` as part of Phase 3.
- SVG fallback (when `svgKey` has no matching pose): `getFallbackSvg()` renders a generic figure. This is acceptable for Phase 3 since Phase 4 populates all exercises. Fallback must be visually distinguishable from a real illustration — it already is (no motion arrow, generic pose).

**Known SVG key mismatch (flagged for Phase 3 implementer):** `exercises.js` stores `svgKey` in camelCase (e.g. `"benchPress"`) but `svgIllustrations.js` uses Spanish kebab-case keys (e.g. `"press-pecho-plano"`). The `getSvg()` function looks up `poses[exerciseId]` directly, so passing a camelCase key will always hit the fallback. Phase 3 must resolve this by one of: (a) updating `svgKey` values in `exercises.js` to match Spanish kebab-case pose keys, or (b) adding camelCase aliases to the poses object. Option (a) is simpler. The planner must include this as an explicit task.

## Component Inventory

### Exercise Card (`.exercise-card`)

One card per exercise. 6 cards rendered per session. Vertically stacked, full-width.

**Structure (top to bottom):**
1. SVG illustration container — full card width, aspect ratio 400:280 maintained via `aspect-ratio: 400/280`
2. Card body — padding `var(--space-lg)` (24px) all sides
   a. Muscle group tag (`.exercise-group-tag`) — pill shape, `--radius-pill`, inline-block above the name
   b. Exercise name (`<h2>`) — Crimson Pro 20px weight 300
   c. Description (`<p>`) — Inter 16px weight 400, line-height 1.6
   d. YouTube button — see component below

**Card CSS contract:**
```
background-color: var(--color-cream)
border: 2px solid var(--color-teal)
border-radius: var(--radius-card)   /* 12px */
overflow: hidden                     /* clips SVG illustration corners */
width: 100%
```
No `box-shadow`. No `background` gradient on the card itself.

## Interaction Contract

### Regenerate
- **Trigger:** Click/tap `.regenerate-btn`
- **Action:** Call `routine.build()` to produce a new 6-exercise array (with recency exclusion applied), re-render `.exercise-list` in-place
- **localStorage key:** `gymup_recent_{routineId}` — stores array of last 6 exercise IDs shown for that routine
- **Recency exclusion rule:** Before picking, filter out any exercise whose ID appears in the stored array. If the filtered pool has fewer than `count` exercises, fall back to the full pool (prevents deadlock with small libraries in Phase 3).
- **Feedback:** No animation in Phase 3 (staggered card animation is Phase 5). The in-place re-render provides implicit feedback.
- **Data attribute on button:** `data-action="regenerate"` — picked up by event delegation in `app.js`

### YouTube Button
- **Trigger:** Click/tap `.youtube-btn`
- **Action:** Browser opens `https://youtu.be/{videoId}` in a new tab (`target="_blank"`)
- **No JavaScript needed** — this is a plain `<a>` tag
- **Accessibility:** `rel="noopener noreferrer"` required. `aria-label="Watch {exercise name} on YouTube"` required.

### Back Button (existing — no changes)
- Already implemented in Phase 2. No changes in Phase 3.
- `data-action="back"` already handled by event delegation in `app.js`.

### Focus Management
- When Regenerate is tapped, focus stays on the Regenerate button after re-render (do not move focus to first card — disorienting on mobile)
- All interactive elements respond to `:focus-visible` with `outline: 3px solid var(--color-teal); outline-offset: 3px`

## localStorage Contract

| Key | Value Type | Content |
|-----|-----------|---------|
| `gymup_recent_day1` | JSON string | Array of up to 6 exercise IDs (strings) shown in last Arms/Shoulders+Abs session |
| `gymup_recent_day2` | JSON string | Array of up to 6 exercise IDs shown in last Chest+Abs session |
| `gymup_recent_day3` | JSON string | Array of up to 6 exercise IDs shown in last Back+Abs session |
| `gymup_recent_day4` | JSON string | Array of up to 6 exercise IDs shown in last Legs session |

Key naming convention: `gymup_recent_{routineId}` where `routineId` matches the `id` field in `workoutDays` (day1, day2, day3, day4).

Behavior on `localStorage` unavailable (e.g. private browsing that blocks storage): catch the `SecurityError`, log a warning, and proceed without recency exclusion. Do not crash.

**Source:** REQUIREMENTS.md (ROUT-06), js/app.js (workoutDays ids)

## Accessibility Contract

- All interactive elements (`<button>`, `<a>`) have `min-height: 48px` (DSNG-06)
- YouTube `<a>` has `aria-label="Watch {exercise name} on YouTube"`
- SVG illustrations have `role="img"` and `aria-label="Exercise illustration"` (updated from Spanish)
- Muscle group tags are presentational — `aria-hidden="true"` if the group information is already conveyed by the exercise name context; otherwise leave visible
- Keyboard navigation order per card: SVG (non-interactive, skip) → exercise name (non-interactive) → description (non-interactive) → YouTube button (focusable)
- Regenerate button comes before the card list in DOM order — keyboard users reach it before scrolling through all cards

**Source:** REQUIREMENTS.md (DSNG-06), CLAUDE.md (Accessibility first)

## Phase 3 Success Verification Checklist

Derived from ROADMAP.md success criteria for Phase 3:

1. Selecting any routine tile renders exactly 6 exercise cards (count verified by DOM inspection: `document.querySelectorAll('.exercise-card').length === 6`).
2. Each card contains: an `<svg>` element, an `<h2>` with the exercise name, a `<p>` with the description, and an `<a class="youtube-btn">` with a valid `href` starting `https://youtu.be/`.
3. Tapping Regenerate produces a visually different set of cards (at least one card differs — verifiable by comparing exercise names before and after with a library of 3+ exercises per group).
4. Running the same routine twice in separate page loads: the second session's 6 exercise IDs do not overlap with the first session's 6 (localStorage recency exclusion confirmed working).
5. All 6 YouTube thumbnail images load (Network tab shows 200 responses from `img.youtube.com`).
6. Regenerate button and all YouTube buttons have min-height 48px (verified in DevTools Computed Styles).
7. No `box-shadow`, `linear-gradient` on card or button CSS (DevTools Computed Styles confirm).
8. SVG labels read "START" and "END" (not Spanish "INICIO" / "FIN").
9. Layout has no horizontal scroll on a 375px viewport (Chrome DevTools device simulation).
